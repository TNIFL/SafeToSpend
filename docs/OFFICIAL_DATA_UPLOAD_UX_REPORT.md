# 공식 자료 업로드 UX 보고서

## 문제 요약
- 홈택스와 NHIS는 메뉴가 깊어서 사용자가 공식 자료를 찾다가 쉽게 헤맨다.
- 자료를 왜 올려야 하는지 설명이 없으면 "왜 또 올려야 하죠?"라는 반응이 나온다.
- 업로드 UX와 개인정보 안내가 분리되면 기대 관리가 실패한다.

## UX 원칙
- 결과를 먼저 보여주고, 정확도를 더 올릴 때만 공식 자료를 요청한다.
- 공식 사이트 바로가기와 메뉴 경로 텍스트를 같이 보여준다.
- "처음 한 번 보정 + 이후 자동 관리 + 필요 시 재확인" 구조를 유지한다.
- `영구 자동화`, `한 번 올리면 끝` 같은 표현은 쓰지 않는다.

## 기본 플로우
1. 왜 필요한지 설명
   - 예: `이 자료를 올리면 세금/건보료 숫자를 더 실제에 가깝게 맞출 수 있어요.`
2. 공식 사이트 바로가기
   - 홈택스/NHIS 공식 사이트 버튼 제공
3. 실패 시 메뉴 경로 안내
   - 버튼이 안 열리거나 메뉴가 바뀌었을 때 텍스트 경로 표시
4. 업로드 후 좋아지는 값 안내
   - 세금 정확도, 건보료 정확도, 증빙 정리 품질 중 무엇이 좋아지는지 표시
5. 기준일 표시
   - `2026-03-15 기준 확인 자료`처럼 스냅샷 시점을 함께 보이기
6. 필요 시만 재확인 요청
   - 새 시즌, 숫자 급변, 기준일 경과 시만 다시 요청

## 자료 요청 시점
### 세금 정확도 업그레이드
- 원천징수·지급명세·납부내역
- 사업용 카드 사용내역
- 전자세금계산서 내역

### 증빙 정확도 업그레이드
- 현금영수증 지출증빙 내역
- 사업용 카드 사용내역
- 전자세금계산서 내역

### 건보료 정확도 업그레이드
- 보험료 납부확인서
- 자격득실확인서

## 서비스 내 진입 위치
- overview: `공식 자료로 숫자 보정하기` 진입점
- 세금보관함: `홈택스 자료 가져오기 안내` 진입점
- 건보료 화면: `NHIS 자료 가져오기 안내` 진입점
- 가이드 페이지: `/guide/official-data`

## 대표 카피 원칙
- `처음 한 번만 맞춰 두면, 이후에는 바뀐 숫자만 다시 확인할게요.`
- `새 시즌이 오거나 숫자가 크게 달라질 때만 다시 확인해요.`
- `원본 파일을 항상 저장하는 방식이 아니라, 필요한 값만 먼저 정리하는 구조를 우선 검토하고 있어요.`

## 사용자 불만 포인트와 대응 문구
### 왜 또 올려야 하죠?
- `매번 다시 받는 구조가 아니라, 기준일이 지난 자료나 숫자가 크게 달라진 자료만 다시 확인해요.`

### 이 자료는 왜 필요한가요?
- `이 자료가 있으면 추정이 아니라 공식 기준에 가까운 숫자로 맞출 수 있어요.`

### 올리면 뭐가 좋아지나요?
- `예상세금, 건보료, 증빙 정리 상태가 더 실제 값에 가깝게 맞춰져요.`

## 업로드 전/후 화면 메시지 방향
- 업로드 전: 왜 필요한지, 무엇이 좋아지는지, 저장 방식 요약
- 업로드 후: 기준일, 자동 관리 범위, 다시 확인할 수 있는 조건

## 이번 설계에서 실제로 맞춘 범위
### A. 공식 자료 대상/공식 경로
- 상세 기준은 `docs/OFFICIAL_DATA_SOURCE_GUIDE.md`를 source of truth로 둔다.

### B. 공식 자료 업로드 UX 구조
- 서비스 안에서는 `/guide/official-data`를 중심 허브로 둔다.
- overview / 세금보관함 / 건보료 화면에서 같은 허브로 들어가게 한다.

### C. 저장 단위 정책
- 상세 기준은 `docs/OFFICIAL_DATA_STORAGE_POLICY.md`, `docs/OFFICIAL_DATA_RETENTION_POLICY.md`를 따른다.
- 기본안은 `추출값 우선 저장 + 원본 기본 비저장 또는 선택 저장 검토`다.

### D. 동의/정책 정합
- 상세 기준은 `docs/OFFICIAL_DATA_CONSENT_MAP.md`, `docs/PRIVACY_POLICY_UPDATE_NOTES.md`, `docs/TERMS_AND_PRIVACY_ALIGNMENT.md`를 따른다.
