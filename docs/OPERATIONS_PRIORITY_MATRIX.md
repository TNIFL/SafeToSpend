# Operations Priority Matrix

작성일: 2026-03-11  
목적: 운영 준비 항목 우선순위 3단 분류

## 우선순위 분류표

| 항목 | 우선순위 | 이유 |
|---|---|---|
| 결제/구독 복구 도구(order/payment/event/reproject) | 오픈 전 필수 | 결제 장애 시 즉시 복구 불가하면 금전/신뢰 리스크가 큼 |
| 문의 채널(인앱 + 대체 채널) | 오픈 전 필수 | 앱 장애 시 사용자 소통 단절 방지 필요 |
| 정책 문서(약관/개인정보/자동결제/해지) | 오픈 전 필수 | 구독형 서비스의 법적/신뢰 기준선 |
| DB 백업 + 복구 리허설 | 오픈 전 필수 | 데이터 유실/장애 복구의 최소 조건 |
| 장애 알림(500/DB/결제 실패/배치 실패) | 오픈 전 필수 | 1인 운영에서 조기 인지가 핵심 |
| 데이터 정합성 일일 점검 | 오픈 전 필수 | 조용한 오작동 방지(결제/증빙/계좌) |
| 스테이징 실측(registration/checkout/webhook) | 오픈 전 필수 | 코드 PASS만으로 오픈 판단 불가 |
| 인프라 로그/APM/CSP 실측 | 오픈 전 필수 | 민감값 노출/CSP 차단은 코드 밖에서 발생 |
| 운영 공지 템플릿 정착 | 오픈 직후 1~2주 내 | 사고 시 안내 품질 향상 |
| 운영 지표 임계치 자동화 | 오픈 직후 1~2주 내 | 수동 모니터링 의존도 축소 |
| 계좌 동기화/패키지 재처리 도구 | 오픈 직후 1~2주 내 | 사용자 문의 대응 시간 단축 |
| 관리자 통합 운영 UI | 나중 고도화 | 현재 CLI+문서로 최소 대응 가능 |
| 관리자 2FA/세분화 권한 | 나중 고도화 | 중요하지만 오픈 직전 최소선은 admin guard로 확보 |
| 업로드 악성코드 스캔 파이프라인 | 나중 고도화 | 현재 시그니처/용량 검증은 있으나 고도화 필요 |
| 고급 관측성(분산추적/SLO 대시보드) | 나중 고도화 | 초기 운영 안정화 후 단계적 확대 권장 |

## 필수 항목 판정 메모
- 현재 기준 오픈 전 필수 항목 중 `백업/복구 리허설`, `정책 문서 공개`, `스테이징·인프라 실측`이 미완료다.
- 따라서 실서비스 오픈은 `CONDITIONAL GO`가 아니라 `NO-GO`가 타당하다.
