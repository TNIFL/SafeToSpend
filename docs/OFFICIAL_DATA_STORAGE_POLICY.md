# 공식 자료 저장 정책 초안

## 기본 원칙
- 최소수집: 서비스 목적에 필요한 값만 먼저 저장한다.
- 목적 제한: 세금/건보료 정확도 보정, 증빙 정리, 세무사 전달 보조 범위에서만 사용한다.
- 원본 최소화: 기본은 핵심 추출값 저장, 원본은 선택 저장 구조를 우선 검토한다.
- 민감/고유식별정보 주의: 주민등록번호, 건강 관련 정보, 자격 변동 정보는 별도 주의 대상으로 다룬다.

## 저장 단위 구분
### 1) 핵심 추출값만 저장 권장
- 문서 종류
  - 현금영수증 지출증빙 내역
  - 사업용 카드 사용내역
  - 전자세금계산서 내역
  - 원천징수·지급명세·납부내역
  - 보험료 납부확인서
  - 자격득실확인서
- 저장 예시
  - 기준일
  - 합계 금액
  - 기간
  - 문서 종류
  - 주요 상태값
  - 필요한 식별용 일부 마스킹 값

### 2) 원본 파일 선택 저장
- 1차 정책 초안
  - 홈택스 자료는 사용자가 명시적으로 선택한 경우에만 원본 저장 검토
  - 원본 저장이 필요한 목적은 재확인, 세무사 전달 보조, 파싱 검증으로 제한
- 사용자 설명
  - `기본은 필요한 값만 정리하고, 원본 파일 저장은 직접 선택한 경우에만 검토해요.`

### 3) 원본 비저장/즉시 폐기 우선 검토
- NHIS 자료는 민감/고유식별정보 가능성이 높아 1차 범위에서는 원본 비저장을 기본안으로 둔다.
- 원본이 꼭 필요하지 않다면 추출 직후 폐기하고, 기준일이 있는 핵심 값만 남긴다.

## 자료별 기본안
| 자료 | 기본 저장 단위 | 이유 | 사용자 설명 |
| --- | --- | --- | --- |
| 홈택스 현금영수증/사업용카드/전자세금계산서 | 추출값 기본 저장 | 세금/증빙 보정에 필요한 핵심 값 위주 | `필요한 값만 먼저 정리해 둘게요.` |
| 홈택스 원천징수/지급명세/납부내역 | 추출값 기본 저장 | 이미 빠진 세금 반영과 정확도 보정 중심 | `이미 빠진 세금을 맞추는 데 필요한 값 위주로 봐요.` |
| NHIS 보험료 납부확인서 | 추출값 기본 저장, 원본 기본 비저장 | 민감 가능 정보 최소화 | `건보료 기준일과 핵심 값만 먼저 반영해요.` |
| NHIS 자격득실확인서 | 추출값 기본 저장, 원본 기본 비저장 | 가입유형/자격 변동만 확인하면 되는 경우가 많음 | `자격 변동 시점처럼 필요한 값만 남겨요.` |

## 기준일 스냅샷 개념
- 공식 자료는 `기준일이 있는 스냅샷`으로 다룬다.
- UX에는 `언제 기준 자료인지`, `언제 다시 확인할 수 있는지`를 함께 표시한다.
- 같은 자료라도 기준일이 오래되거나 시즌이 바뀌면 재확인을 요청할 수 있다.
