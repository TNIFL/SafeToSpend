# 회원가입 동의 구조 초안

- 상태: 초안
- 기준: `main` 브랜치 `20624f29f8aa6bae7d9ecf1f6ffaea49374afeb6`
- 목적: 현재 구현 사실과 현재 확정된 정책에 맞는 가입 동의 구조 기준선 정리
- 주의: 이 문서는 운영 확정본이 아니다. 실제 적용 전에는 화면 구조, 동의 로그, 버전 관리, 운영주체 정보가 함께 확정되어야 한다.

## 1. 현재 구현 기준 핵심 전제

- 현재 회원가입 화면은 `이메일`, `비밀번호`, `비밀번호 확인`만 받는다.
- 현재 코드 기준으로 약관 동의 체크박스, 전문보기, 동의 로그 저장은 없다.
- 현재 결제 승인, 정기결제, 플랜별 entitlement는 연결되어 있지 않다.
- 현재 공식자료 자동수집, 원본 선택 보관, 민감정보 별도 동의 흐름은 구현되어 있지 않다.

## 2. 가입 화면 체크박스 구조 권고안

### 지금 바로 넣어야 하는 항목

- `[필수] 이용약관 동의`
- `[필수] 개인정보처리방침 동의`

### 후속 구현과 함께 넣을 수 있는 항목

- `[선택] 마케팅 정보 수신 동의`

### 지금 기본 가입 필수에 넣으면 안 되는 항목

- 민감정보 별도 처리 동의
- 주민등록번호 처리 동의
- 공식자료 원본 선택 보관 동의
- 자동수집 특별 동의
- 건강보험 자격·납부자료 별도 처리 동의

## 3. 왜 이렇게 나누는지

- 이용약관과 개인정보처리방침은 계정 생성과 기본 서비스 이용에 직접 연결되므로 필수 동의로 설계한다.
- 마케팅 정보 수신은 핵심 서비스 제공과 직접 관련이 없으므로 선택 동의로 분리한다.
- 민감정보, 주민등록번호, 원본 선택 보관, 자동수집은 현재 기본 가입 과정에서 즉시 필요하지 않고, 별도 기능 활성화 시점에 별도 동의로 분리하는 편이 법적 리스크가 낮다.
- 선택 동의를 거부해도 가입과 핵심 서비스 이용이 가능해야 한다.

## 4. 화면 구조 권고

- 필수 동의 영역과 선택 동의 영역을 시각적으로 분리한다.
- 각 항목 옆에는 `전문보기` 링크를 둔다.
- 필수 항목은 모두 체크해야 가입 버튼이 활성화되게 한다.
- 선택 항목은 기본 비체크 상태로 둔다.
- `전체 동의`를 넣더라도 필수/선택 구분이 사라지지 않게 한다.

## 5. 즉시 노출 보류 권고

### 마케팅 수신 동의

- 구조상 선택 항목으로는 적절하다.
- 다만 실제 발송 기능, 수신 채널, 철회 방식, 동의 로그 저장이 준비되지 않았다면 가입 화면에 먼저 노출하지 않는 편이 안전하다.

### 민감정보·원본보관 관련 동의

- 현재 구현 범위와 정책이 아직 확정되지 않았으므로 기본 가입 화면에 두지 않는다.
- 특히 건강보험 자격 관련 자료, 주민등록번호 포함 가능성이 있는 원본 자료는 별도 기능과 별도 고지, 별도 동의가 필요하다.

## 6. 후속 구현 체크리스트

- 이용약관/개인정보처리방침 전문 페이지 제공
- 필수/선택 체크박스 UI 구현
- 동의 시각 저장
- 동의 문서 버전 저장
- 변경 고지 이력 관리
- 마케팅 동의 수집 시 수신 채널, 철회 UI, 철회 로그 구현
