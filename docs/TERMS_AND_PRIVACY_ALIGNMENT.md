# UX / 정책 정합 점검표

## 목적
- 업로드 UX 카피와 개인정보처리방침/동의 문구가 서로 어긋나지 않는지 점검한다.

## 정합 기준
| UX 문구 | 정책 문구 | 점검 기준 |
| --- | --- | --- |
| `처음 한 번 보정 + 이후 자동 관리` | `기준일이 있는 공식 자료를 바탕으로 필요한 값만 저장` | 자동 관리 범위를 과장하지 않는지 |
| `새 시즌이나 숫자가 크게 달라질 때만 다시 확인` | `재확인 시점과 기준일 정책 명시` | 재확인 조건이 정책과 일치하는지 |
| `기본은 필요한 값만 저장` | `핵심 추출값 우선 저장, 원본은 기본 비저장/선택 저장` | 저장 단위가 UX와 정책에서 같게 보이는지 |
| `올리면 세금/건보료 숫자가 더 정확해져요` | `공식 자료 분석 목적` | 목적 설명이 과장되지 않는지 |
| `업로드를 안 해도 기본 기능은 사용할 수 있어요` | `선택 동의 / 거부 가능 범위` | 거부 시 영향 범위가 맞는지 |

## 현재 불일치 방지 원칙
- UX에서는 생활 언어를 쓰되, 정책 문서에서 의미가 달라지지 않게 한다.
- `영구 자동화`, `무기한 원본 보관` 같은 표현은 UX와 정책 모두에서 금지한다.
- NHIS 자료처럼 민감 가능성이 있는 문서는 원본 기본 비저장 원칙을 일관되게 유지한다.
